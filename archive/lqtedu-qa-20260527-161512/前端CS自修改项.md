# 前端 CS 自修改项

## AI_FAST_READ

scope:
- 本文记录老板口中的前端 `CS`，也就是写页面、改页面时默认要懂、要主动处理的样式常识。
- 适用于前端页面里的确定性视觉问题：间距、按钮排列、颜色、圆角、日期展示格式。
- 如果 `Base` 页已经有确定控件或确定样式，工作页面里出现不一致，应默认按 `Base` 页自行修正。
- 如果改动牵涉复杂业务逻辑、接口逻辑、权限、数据规则、流程判断，立刻跳过，不要脑补硬改。

core_rules:
- `Base` 页是基础样式参照源；能复用已有控件、颜色、间距规则时，不另造一套。
- RP 提到“新样式”时，优先理解为 `Base` / `EBase` 已有 DOM 或组件样式，不是新增一套 CSS。
- 用户要求“完全复制 Base 页样式”时，先整页或整块复制母版 DOM、class、组件写法，再只改业务参数；不要重写一个“看起来差不多”的版本。
- 接口未到但页面需要先可运行时，优先做页面局部数据适配层和 mock 开关，保留真实接口方法名、入参和返回结构，后期只替换适配层。
- Vue3 + Element Plus 的 JSP 页面里，组件脚本如果先于页面主脚本加载，不要在组件文件顶层重复声明 `ref`、`computed`、`watch`、`onMounted` 等 Vue helper，避免和页面主脚本的 `const { ... } = Vue` 冲突。
- 标题与内容之间：`25px`。
- 内容与底部按钮之间：`25px`。
- 表格上方元素与表格之间：`8px`。
- 各类按钮之间：`5px`。
- 同一个按钮内，图标与文字之间：`3px`；即使组件库或旧样式默认不是 `3px`，也按这条覆盖。
- 同组元素间距优先让父容器用 `display: flex/grid` + `gap` 控制；块级结构距离、RP 固定距离、旧样式兼容或特殊偏移时再用 `margin`。
- 颜色按 `Base` 页；能使用英文颜色名时，使用英文名。
- 所有方框、弹框都要有圆角。
- 弹框及以上的大块容器圆角：`5px`。
- 大方按钮及以下的小控件圆角：`2px`。
- 日期展示格式统一把 `-` 改成 `/`。

hard_stop:
- 不是纯样式问题，而是会影响业务含义、数据处理、接口入参、保存逻辑、权限逻辑时，先停下来确认。
- `Base` 页没有确定样式、截图/RP 与现有页面冲突、或规则本身无法判断时，先问清楚。
- 不因为“看起来差不多”就改公共组件或全局样式；优先局部修当前页面的明确问题。
- 接口未到场时，不要让页面默认打必失败的真实接口；如果用户明确要求先用假数据，假数据必须集中在适配层，不能散落在模板和业务流程里。

## 1. 这份规则怎么理解

这里的 `CS` 不是新需求，而是前端开发时默认要自检、自修的基础规范。也就是说，页面里出现明显和 `Base` 页不一致的控件、间距、颜色、圆角、日期格式，不需要等别人逐条指出，前端应主动修到一致。

但这个“自修改”只限确定性样式。只要碰到复杂业务逻辑，就不要逞能。比如字段含义、状态流转、按钮权限、接口参数、保存规则、数据过滤，这些不是样式常识，不能靠猜。

## 2. 可直接自修改项

### 2.1 以 Base 页为准

- `Base` 页中已经确定的控件样式、颜色、间距、圆角，是默认参照。
- 当前工作页面的空间关系或样式与 `Base` 页不同，且不涉及业务逻辑时，直接修正。
- 已有同类控件能复用时，优先复用，不要重新写一套看起来差不多的样式。烂代码里最怕这种“差不多”，最后全项目都差一点。
- RP 说“新样式”时，先去 `base.jsp` / `ebase.jsp` 找对应 DOM 示例：按钮、文字操作、链接、表格、分页、步骤条、表单控件、弹窗、标签、提示等。
- 找到同类 DOM 后，复用它的结构和 class；不要只抄颜色，也不要自己猜 hover、边框、下划线、间距。
- 只有 `Base` / `EBase` 没有对应样式，或 RP 明确要求了现有样式覆盖不了的差异，才写当前页面局部样式。

### 2.1.1 整块复用与接口未到场

- 如果目标页是某个已有页面的变体，例如 `newexam` 成绩导入参考标准考试成绩导入，先复制源页面和组件文件，再做最小业务适配：参数读取、权限判断、返回链接、默认模式。
- “完全复制样式”指 DOM 结构、组件类型、class、按钮 type、内联布局和提示样式都要跟母版一致；不要自行重排、换控件、换按钮 class 或补一套局部 CSS。
- 真实接口未到场时，用 `USE_MOCK` 这类显式开关和 `xxxApi` 适配对象集中承接下拉、上传、生成模板、确认导入、历史记录等调用。
- mock 数据要贴近后续接口结构：字段名、数组层级、状态字段、历史记录里的 `FILE_STR/title/students` 等结构先按页面消费形态固定，后端接口到场后只替换适配对象内部实现。
- JSP 内 Vue3 页面常见加载顺序是“组件脚本先加载、页面主脚本后声明 Vue helper”。组件文件顶层只声明必要组件入口，例如 `defineComponent`；不要顶层声明 `ref/computed/watch/onMounted`，否则页面主脚本再 `const { ref } = Vue` 会报 `Identifier 'ref' has already been declared`。

### 2.2 基本行距

| 场景 | 间距 |
| --- | --- |
| 标题与内容 | `25px` |
| 内容与底部按钮 | `25px` |
| 表格上方元素与表格 | `8px` |

### 2.3 基本列距

| 场景 | 间距 |
| --- | --- |
| 各类按钮之间 | `5px` |
| 同一个按钮内的图标与文字 | `3px` |

按钮内图标与文字这条只针对“图标 + 文字”的按钮，例如“新增”“查询”这类左侧带图标、右侧带文字的按钮。纯图标按钮、纯文字按钮不受这条影响。

如果项目现有组件库或旧页面默认间距不是 `3px`，也要按 `3px` 覆盖。这里是老板明确说的前端 CS，不是可选建议。

### 2.3.1 `gap` 优先规则

- 同一父容器内的横向或纵向排列，优先使用 `display: flex` / `display: grid` 配合 `gap` 管理间距。
- 按钮组、标签组、行内图标文字、表单内并排项、弹层底部操作区等，能用 `gap` 就不要给每个子元素堆 `margin`。
- 页面结构距离仍按语义处理：标题到内容、内容到底部按钮、表格上方元素到表格这类 RP 固定距离，可以继续用 `margin` 或现有公共类。
- 旧页面兼容、单个元素特殊偏移、公共样式无法调整、或 RP 明确要求某个元素独立位置时，才使用局部 `margin`。
- 不要为了一个 `5px` 间距给一串按钮逐个写 `margin-right`，这类代码后面一改顺序就炸。

### 2.4 颜色

- 颜色按 `Base` 页现有规范走。
- 能使用英文颜色名时，使用英文名。
- 项目里已有颜色 token、class、变量时，优先用现成的，不要手写一堆新的十六进制颜色。

### 2.4.1 样式职责边界

- 颜色、按钮、链接、表格、分页、步骤条、表单控件等，优先走 `Base` / `EBase` 现有规范。
- 当前页只写 RP 明确要求且 `Base` / `EBase` 覆盖不了的局部差异，比如特定间距、特定占位或特定局部排列。
- 不要把“当前旧页面没有”误判成“项目没有”，先查基础页和同类页面。

### 2.4.2 位置不动要看场景

- “位置不动”不能一刀切，要先判断 RP 要求的是相邻元素隐藏不影响其他元素，还是整个操作区布局固定。
- 如果相邻按钮、链接、操作项因为状态、权限、数据原因隐藏，但其他操作项位置不能跳，可以考虑占位隐藏。
- 老 JSP / Avalon 页面里，占位隐藏可用 `visibility`，例如 `:css="{visibility: condition ? 'visible' : 'hidden'}"`；它保留 DOM 和占位。
- 如果 RP 要求的是整列对齐、按钮组固定区域、左右分布或固定宽度操作区，再考虑列宽、按钮容器、inline-block 宽度或局部布局样式。
- 不要一看到“隐藏”就用 `:if` / `ms-if` 删除 DOM，也不要一看到“位置不动”就手写 flex。

### 2.5 圆角

- 所有方框、弹框都应有圆角。
- 弹框及以上的大块容器：`5px`。
- 大方按钮及以下的小控件：`2px`。
- 如果已有同类组件明确写了圆角，以同类组件和 `Base` 页为准，保持一致。

### 2.6 日期格式

- 日期展示格式统一把 `-` 改成 `/`。
- 示例：`2026-04-29` 应展示为 `2026/04/29`。
- 注意这里只记录前端展示格式；如果接口入参或后端存储要求使用 `-`，不能擅自改接口数据格式。

## 3. 不能自作主张的情况

- 改动会影响接口字段、入参、返回值解析。
- 改动会影响按钮是否显示、是否可点、点击后走什么流程。
- 改动会影响数据筛选、排序、分页、权限、状态判断。
- 改动需要改公共组件、公共样式或多个页面联动，但没有明确依据。
- `Base` 页、RP、截图、现网页面之间互相冲突。

遇到以上情况，先停下来问清楚，别用“常识”去补业务逻辑。样式可以有 CS，业务不能靠玄学。

## 4. 开发自检清单

- 当前页面是否已对照 `Base` 页的同类控件？
- 标题与内容是否保持 `25px`？
- 内容与底部按钮是否保持 `25px`？
- 表格上方元素与表格是否保持 `8px`？
- 按钮之间是否保持 `5px`？
- 按钮内图标与文字是否保持 `3px`？
- 同组元素是否优先用父容器 `gap` 管间距，而不是给每个子元素堆 `margin`？
- 颜色是否复用 `Base` 页或项目已有规范？
- 方框、弹框、按钮圆角是否符合 `5px` / `2px` 规则？
- 日期展示是否把 `-` 统一为 `/`？
- 本次修改是否完全停留在样式和展示层，没有偷偷改业务逻辑？
