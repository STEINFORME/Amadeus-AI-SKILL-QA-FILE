## 1. 验证顺序

每次测试默认按这个顺序走：

1. 入口
2. 参数
3. 显隐逻辑
4. 交互状态
5. 视觉细节

别一上来就盯颜色和间距。入口没通、参数不对，后面全是伪结论。

### 1.1 测试前先分清自己在做哪类测试

- `setup`：准备环境、账号、登录态；不输出业务通过/失败结论。
- `smoke`：验证入口和最小可用链路是否存活；默认只看“能不能打开、能不能进入、能不能继续”。
- `regression`：验证具体业务规则或历史缺陷点是否复发。
- `功能/验收`：按 RP 和业务主链路确认需求是否实现。
- `接口/性能/安全`：不归这份页面测试规则主流程；不要混写进页面测试结论。

### 1.2 黑盒、白盒、灰盒先别叫乱

- `黑盒`：不依赖内部实现；仅基于输入、操作、输出下结论。
- `白盒`：依赖内部实现；按代码分支、条件判断、异常路径验证。
- `灰盒`：知道部分实现，但结论仍落在业务行为；前端 mock 接口、本地状态驱动 UI 验证默认归这里。
- 页面测试默认优先按 `黑盒/灰盒` 执行；除非用户明确要求，不主动扩展成白盒结论。

### 1.3 受控流程新增验证项

在入口、参数、显隐、交互、视觉之外，按任务风险补这三类验证：

1. 安全验证：
   - `v-html` / `innerHTML` / JSP 变量直接输出。
   - URL 参数拼接、直接跳转、下载地址拼接。
   - 只隐藏按钮但仍能触发接口。
   - token、手机号、身份证、账号是否进入 URL、console、localStorage/sessionStorage。
   - 上传下载是否有限制、失败处理和越权风险。
2. 异常数据验证：
   - `null`、`undefined`、空数组、空字符串、超长文本、特殊字符、HTML 字符串。
   - 数字字段返回字符串、字符串字段返回数字、日期为空或格式异常。
   - 接口错误码、超时、返回登录页 HTML、无权限、部分成功。
   - `total` 与 `list` 不一致、分页越界、图片失败、文件链接失效。
3. 并发和交互边缘验证：
   - 重复点击提交。
   - 保存中关闭弹窗。
   - 请求未完成时切换筛选条件。
   - 后发请求先回时，旧数据是否覆盖新数据。
   - 删除当前页最后一条后分页是否合理。
   - 编辑后取消是否恢复原数据。
   - 弹窗反复打开是否残留旧数据或校验状态。
   - 保存成功是否刷新列表，保存失败是否保留用户输入。

### 1.4 结论门禁

- `通过`：关键证据至少 A/B 级，字段/状态/权限来源明确，无未解决安全风险，真实 diff 已审查，正常路径、异常数据、权限场景、重复操作不影响结论。
- `受限完成`：页面已完成但仍依赖 Mock、状态枚举/权限字段未最终确认、仅测试环境验证、部分异常场景未验证且已说明风险。
- `未验证`：没有运行页面、没有真实接口、没有可用账号、没有权限验证、没有真实 diff 审查或没有验证关键交互。
- 存在关键未验证项时，不允许写 `通过`。

## 2. 视觉异常排查流程

### 2.1 用户说“附近有异常”，别只盯组件本体

第一轮排查范围至少包括：

- 组件本体
- 父容器
- 相邻元素
- 分割线
- `:focus`
- `:active`
- `border`
- `outline`
- `box-shadow`

### 2.2 看见黑线、蓝边、阴影、发灰、发虚，先按异常类型查

优先排查：

- `border`
- `outline`
- `box-shadow`
- `:focus`
- `:active`
- 父容器边框
- 伪元素

### 2.3 先分默认态和交互态

每次都问自己：

- 默认加载就有吗？
- hover 才出现吗？
- active 才出现吗？
- focus 才出现吗？

不分状态去查，基本就是瞎查。

## 3. 结论和记录规则

### 3.1 结论必须分层

- `通过`
- `有问题`
- `受限完成`
- `非本次范围`
- `未验证`

别写成一锅粥式的流水账。

### 3.2 被阻断时，先停在入口层

这些都属于阻断，不要继续硬写视觉结论：

- 登录进不去
- 页面直接 500
- 缺关键参数
- 依赖接口没通

### 3.3 只读测试边界写死

- 需要真实保存、删除、上传、提交才走得通的链路，默认停在提交前
- 统一记法：
  - `未验证：需提交`
  - `未验证：需造数`

别为了补一句“已验证”去污染线上数据。

## 4. 交互和视觉的测试口径

### 4.1 hover 区要测真实 hover

- RP 写的是移入展开，就要测真实 hover 行为
- 不能拿常驻展开状态替代

### 4.2 hover 里有确认框，要测 hover 锁定

- 打开 `popconfirm` 后，原 hover 区不能瞬间消失

### 4.3 `ElMessage` 不作为默认正确实现

- Vue3 + Element Plus 页里，看见 `ElMessage` 不要默认当成“提示已做对”
- 当前项目默认口径还是 `ElNotification`；除非 RP、参考页或既有实现明确要求，否则 `ElMessage` 基本不使用

### 4.4 计数文案要测细节，不只测有无

- 数字是否变色
- 括号是否跟着变色
- 0 值时是否还保留括号

### 4.5 空态不要自己补测不存在的文案

- RP 没要求空态，不要把“没显示暂无记录”当 bug

### 4.6 单行点击和单链接点击要分开验

- 整行热区要测整行
- 文字入口要测文字点击区

## 5. 数字中枢测试专项

### 5.1 数字中枢不能裸测

- 不能直接双击或地址栏裸开 `web/digitalHub/*.html`
- 必须从 `web/department/office/campus/DataHub/*.jsp` 的“启动页面”进去

### 5.2 数字中枢测试前先确认三件事

1. 有没有 `school_id`
2. `school_id` 是不是入口页生成的加密值
3. 当前测试的是管理页配置，还是静态大屏展示

### 5.3 数字中枢的验证顺序

1. 先确认入口链路
2. 再确认 `school_id`
3. 再看接口是否返回
4. 最后才看图表、轮播、头像、商品、视觉

### 5.4 数字中枢常见误判

- 裸开静态页看到空白，就误判前端挂了
- 忘了 `school_id` 是加密值，拿明文去调接口
- 看到 `/digitalHub/*` 就以为全都在 `DigitalHubService`
- 把 demo 图、测试页、旧目录当成生产主链路

## 6. 常见测试误判

### 6.1 把开发批注当页面文案

- 结果：把本不该显示的说明文字当成缺失项

### 6.2 没先判同构就拿参考页硬比

- 结果：明明结构不同，还硬说视觉不一致

### 6.3 只盯组件，不看周边容器

- 结果：根因在边框或相邻元素，你却越改越偏

### 6.4 把后端驱动块当成静态内容去验

- 结果：接口没通时，测试结论全失真

## 7. RP 可测性和 E2E 落点

### 7.1 先判断 RP 到底能支撑到什么测试深度

- 若 RP 仅提供静态页面，且缺少交互说明、字段规则、异常态、权限说明、接口约定，则测试深度限制为：
  - 页面展示
  - 主流程
  - 基础异常流
  - 必填/格式/显隐
- 遇到缺失规则时，统一输出 `未验证 / 需补规则`。
- 禁止基于猜测补业务规则、补状态机、补接口语义。
- RP 足够完整时，才继续拆回归用例、自动化用例、接口断言。

### 7.2 手工、E2E、单元测试各管什么

- `手工测试`：用于探索性验证、视觉验收、需求不稳定页面。
- `E2E`：用于高频主流程、跨页面链路、历史高风险回归点。
- `单元测试`：用于可抽离规则、状态转换、参数拼装、纯函数逻辑。
- 默认分工：
  - 规则正确性优先单元测试
  - 主流程正确性优先 E2E
  - 视觉与探索性问题优先手工测试

### 7.3 新增 E2E TypeScript 文件放哪

- 当前独立 E2E 工作目录：`D:\software\e2e`
- 当前 E2E 结构说明文件：`D:\software\e2e\E2E_STRUCTURE.md`
- 当前兼容入口示例：`D:\software\e2e\tests\reg-step12.spec.ts`
- 当前 E2E 环境映射：
  - `gz` = 测试云
  - `www` = 正式云 1
  - `w2` = 正式云 2
  - `w3` = 正式云 3
  - `w4` = 正式云 4
- 开发阶段默认禁止使用 `www / w2 / w3 / w4`。
- 开发阶段默认只允许：
  - 本地环境
  - `gz` 测试云
- 除非用户明确要求并确认影响，否则不要在开发阶段执行正式云 E2E、正式云联调、正式云数据验证。
- `tests/smoke/*.spec.ts`：只放冒烟测试。
- `tests/<业务域>/*.spec.ts`：放新的业务测试入口文件。
- `support/<业务域>/*.page.ts`：放页面定位器、页面动作、页面状态读取。
- `support/<业务域>/*.mocks.ts`：放 mock 接口和假响应。
- `support/<业务域>/*.data.ts`：放测试数据工厂和输入构造。
- `auth/*.setup.ts`：放登录态或环境准备。
- `tests/*.spec.ts` 根目录默认不再新增新文件；只有兼容旧命令时，才保留一个薄入口去 `import` 真正的场景文件。
- 新增页面如果叫 `reg-step13`，默认按这个落：
  - `tests/parent-registration/reg-step13.spec.ts`
  - `support/parent-registration/reg-step13.page.ts`
  - `support/parent-registration/reg-step13.mocks.ts`
  - `support/parent-registration/reg-step13.data.ts`
- `*.mocks.ts` 和 `*.data.ts` 不是强制件，页面不需要就别硬建空壳文件。
- 默认执行规则：
  - 新业务测试入口文件一律放 `tests/<业务域>/`
  - 新页面动作抽到 `support/<业务域>/*.page.ts`
  - 重复 mock 超过 2 处再抽 `*.mocks.ts`
  - 重复测试数据构造超过 2 处再抽 `*.data.ts`

## 8. 提交前自查清单

- 入口能进吗？
- 参数对吗？
- 关键显隐逻辑跑了吗？
- 交互态都测了吗？
- 视觉异常有没有按状态拆开查？
- 结论有没有明确标 `通过 / 有问题 / 非本次范围 / 未验证`？
- 未验证项有没有写清楚阻断原因？
- 数字中枢如果有问题，是否先确认过 `school_id` 和启动路径？
