# 热修流程说明

## AI_FAST_READ

scope:
- 本文只记录前端项目分支走热修时的 Git 流程。
- 示例以项目分支 `912`、热修分支 `hotfix/260508` 为准。
- 热修分支必须从最新 `master` 建，不要从项目分支建。
- 项目分支推送或确认已和远端一致后，必须重建部署包目录：清空仓库根目录 `deploy/`，再把项目分支相对 `origin/master` 改动的前端文件按原路径放入 `deploy/`，例如 `web/oa/data/dataManager.jsp` 放到 `deploy/web/oa/data/dataManager.jsp`。
- 热修完成后的提醒对象是项目分支：提醒发布人将远端项目分支合并进 `master` 后删除项目分支，例如删除 `origin/912`。

hard_stop:
- 本地有未提交改动、未跟踪文件时，先确认是否与热修有关；无关文件不要带进热修。
- `master` 和 `origin/master` 不一致时，先同步清楚再建热修分支。
- 项目分支和远端项目分支不一致时，先确认差异，不要直接脑补。
- 清空 `deploy/` 属于删除操作，执行前必须确认目标路径是当前仓库根目录下的 `deploy`，且必须得到用户明确确认。
- `deploy/` 只放本次热修要交付的前端文件，默认只收 `web/**`；后端、配置、删除类变更不要假装成前端部署文件，需单独说明。
- 涉及 `git push`、远端删除、强制覆盖、`reset --hard` 等危险操作，必须先取得用户明确确认。

## 1. 适用场景

项目分支已经完成修复，需要按热修流程发布。例如：

- 项目分支：`912`
- 热修分支：`hotfix/260508`
- 修复内容：`头像框默认头像修复`

## 2. 标准流程

1. 确认当前项目分支已经推送到远端。

   本地 `912` 必须和 `origin/912` 对齐。若本地分支有 ahead 提交，需要先推送项目分支。

2. 重建 `deploy/` 部署包目录。

   在项目分支远端已确认后，清空仓库根目录 `deploy/`，再根据 `origin/master..origin/912` 的前端变更清单，把文件从 `origin/912` 按原始路径导出到 `deploy/`。例如源文件 `web/space/common/data/dataManager.jsp`，部署包路径就是 `deploy/web/space/common/data/dataManager.jsp`。

   注意：这里清空的是 `deploy/` 目录内容，不是删除仓库文件。执行前必须确认路径，别把这破项目的其他目录误删了。

3. 从最新 `master` 建立热修分支。

   热修分支命名为 `hotfix/日期简述`，日期通常用 `YYMMDD`，例如 `hotfix/260508`。

4. 推送热修分支到远端。

   先把从 `master` 新建出来的 `hotfix/260508` 推到远端，保证远端有这个热修分支。

5. 将远端项目分支合并到热修分支。

   在本地 `hotfix/260508` 上合并 `origin/912`，再推送 `hotfix/260508`。

6. 提醒发布人后续处理。

   正确提醒是：请发布人在远端将 `912` 合并进 `master`，完成后删除远端 `912` 分支。

   注意：这里删除的是项目分支 `912`，不是默认删除 `hotfix/260508`。除非发布规则另有要求或用户明确要求，否则不要把删除 `hotfix` 分支当成默认提醒。

## 3. 命令模板

```bash
# 先同步远端引用
git fetch origin

# 检查当前状态，不能夹带无关本地改动
git status --short --branch

# 检查 master 是否最新
git rev-parse master
git rev-parse origin/master

# 检查项目分支是否已推远端
git rev-parse 912
git rev-parse origin/912

# 查看项目分支相对 master 的提交，确认没有夹带脏提交
git log --oneline --decorate master..912
git log --oneline --decorate 912..master

# 重建 deploy 部署包目录
# 清空 deploy 是删除操作，执行前必须先确认
$projectBranch = "912"
$baseRef = "origin/master"
$headRef = "origin/$projectBranch"
$repoRoot = git rev-parse --show-toplevel
$deployDir = Join-Path $repoRoot "deploy"
New-Item -ItemType Directory -Force -Path $deployDir | Out-Null
$expectedDeploy = [System.IO.Path]::GetFullPath($deployDir).TrimEnd('\')
$resolvedDeploy = [System.IO.Path]::GetFullPath((Resolve-Path $deployDir).Path).TrimEnd('\')
if ($resolvedDeploy -ne $expectedDeploy) { throw "deploy 路径异常，停止清空" }
Get-ChildItem -LiteralPath $resolvedDeploy -Force | Remove-Item -Recurse -Force
$files = @(git diff --name-only --diff-filter=ACMRT "$baseRef..$headRef" -- "web")
if ($files.Count -gt 0) {
  $zipPath = Join-Path $env:TEMP "lqtedu-deploy-$projectBranch.zip"
  if (Test-Path $zipPath) { Remove-Item -LiteralPath $zipPath -Force }
  git archive --format=zip -o $zipPath $headRef -- $files
  Expand-Archive -LiteralPath $zipPath -DestinationPath $resolvedDeploy -Force
  Remove-Item -LiteralPath $zipPath -Force
}

# 从 master 创建热修分支
git switch -c hotfix/260508 master

# 推送热修分支到远端
git push -u origin hotfix/260508

# 合并远端项目分支
git merge origin/912

# 推送合并后的热修分支
git push origin hotfix/260508
```

如果 `912` 还没有推到远端，先在 `912` 分支上执行：

```bash
git push -u origin 912
```

## 4. 验证清单

- `master` 和 `origin/master` 指向一致。
- 项目分支和远端项目分支指向一致，例如 `912 == origin/912`。
- `deploy/` 已清空并重新生成，只包含 `origin/master..origin/912` 中改动的前端文件，且路径保持为 `deploy/web/...`。
- 热修分支从 `master` 创建，不是从 `912` 创建。
- 合并后 `hotfix/260508`、`origin/hotfix/260508` 都包含 `origin/912` 的修复提交。
- `git status --short --branch` 没有出现本次热修不该携带的已暂存或已修改文件。
- 最终提醒写清楚：远端 `912` 合并进 `master` 后删除远端 `912`。

## 5. 912 示例结论

以 `912` 为例，完整结论应写成：

```text
origin/912 已经合并进 origin/hotfix/260508。
接下来请发布人在远端将 912 合并到 master，完成后删除远端 912 分支。
```

不要写成：

```text
发布后删除 hotfix/260508。
```
