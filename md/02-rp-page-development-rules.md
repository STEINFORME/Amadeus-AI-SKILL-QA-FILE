# RP 页面开发规则摘要

## 适用范围

用于 LQTedu 前端页面开发前判断：页面归属、参考实现、技术栈、交互边界、字段契约、Mock 边界、影响面和安全风险。

完整规则来源于原 `开发前必读-RP页面开发规则.md`，已迁移为本目录长期知识。

## 默认流程

1. 先读任务和入口，不直接写代码。
2. 确认证据：RP、截图、设计稿、现网页面、真实接口、已有实现。
3. 给证据分级：S/A 可直接作为实现依据，B 级可参考但要核对，C/D 级只能当线索。
4. 判断页面壳层和技术栈：旧 JSP/Avalon、Vue3/Element Plus、独立页、后台配置页。
5. 判断 RP 和参考页是否同构：结构、标题、分层、按钮位置、交互位置都要对齐。
6. 查现成实现：`base.jsp` / `ebase.jsp` -> 同模块 -> 同技术栈同类型页面。
7. 列字段契约：请求参数、返回字段、状态枚举、权限字段、跳转参数都写清来源。
8. 先做影响面和安全检查，再做最小必要改动。
9. 修改后复查真实 diff、文件内容、规则和验证结果。

## 分支职责

- 前后端分离项目的前端需求只开或切 `项目名-fe` 分支，例如 `896-fe`。
- `项目名-fe` 表示前端职责边界，不带 `codex/` 前缀。
- 不主动开后端分支，不把前端需求扩大成后端改动。
- 发现后端问题时，只说明接口、字段、权限、状态或数据契约影响。

## 和 skill 的关系

- 开发任务调用 `lqtedu-dev-workflow` Skill 时，完整角色编排、subagent 创建、授权暂停、Developer 分工、Reviewer 和 Reporter 输出以 Skill 为准。
- 使用 Skill 只表示允许真实 subagent 做只读探索、交叉核对和方案编排，不等于授权修改代码。
- 是否进入实现阶段，以 Skill 中 Solution Planner 的授权暂停规则为准。
- 本目录只沉淀开发判断规则和项目经验，不重复维护完整人员安排流程。

## 硬性暂停条件

遇到以下情况先停：

- 页面依赖字典、部门树、人员树、权限、菜单、保存接口，但后端边界不清。
- 技术栈、入口、脚本生命周期、JSP include、全局变量来源不清。
- 接口字段、状态枚举、权限字段、跳转参数只能靠字段名或旧页面猜。
- 设计稿只有正常态，但任务涉及空态、加载态、错误态、禁用态、权限差异态或弹窗状态。
- 要改公共组件、全局 CSS、公共请求、公共权限、全局菜单、路由或 include。
- 涉及删除、提交、发布、审核、撤回、批量、导入导出、上传下载、富文本、`v-html`、`innerHTML`。
- 当前分支、目标环境、上线范围或测试云范围不清。
- 准备直接套模板，但不能证明同构。
- RP 只有静态图，没有交互、接口或权限说明。
- 公共 include 已有 `ROOTPath`、`param`、`schoolName`，页面里不要重复声明。

## 实现路由

- 旧 JSP / 原生 / Avalon：优先查 `base.jsp` 和旧 CSS class。
- Vue3 + Element Plus：优先查 `ebase.jsp`，提示默认使用 `ElNotification`，不要随手换成 `ElMessage`。
- 颜色：优先 `var(--mainBlue/mainGreen/mainGray/mainRed/lineDisable)` 或 base 色板。
- 表格：RP 未明确列宽不要写死；整行点击就做行热区，不硬套 `el-link`。
- 表格操作列：如果某个操作隐藏后位置不能跳动，保留 DOM 占位，用 `visibility: hidden` 一类方式隐藏。
- Vue3 表单：RP 上红色 `*` 必须落到 `el-form` / `el-form-item` 的 `prop + rules`，提交触发表单校验。
- 日期：页面展示统一 `/`，提交前按接口契约转成 `-`。
- 弹层：小确认优先 `el-popconfirm`；Bootstrap 页面不要强行换 Element Dialog。
- 上传和第三方组件：先看浏览器最终 DOM，再写样式。

## 班主任手册开发口径

- 班主任手册挂在“我的班级”左栏，不是独立系统入口。
- 左栏入口落 `body-left.jsp` 对应分支，页面传正确 `top-active`、`dept-left`。
- 工作导览、派发、添加、查看、汇总、详情、检查要按同组页面整块对齐。
- 单次派发不显示批次区，多批次派发是文本串，右侧操作语义是“取消后续派发”。
- 个人派发列表状态看 `SUCCESS_STATUS`，不是 `OVER_TIME`。
- 检查页已完成记录字段看 `updateList.details`，不是顶层 `details`。
- 添加、查看、检查的动态字段编辑态统一使用 `TEXT_TYPE + el-form` 规则。
- 个人派发进入添加页只传 `indexId`，不要把派发列表 `ID` 当个人记录 `actId`。

## 交付前自查

- 设计依据是否完整。
- 关键证据是否至少 A/B 级。
- 页面属于 `base` 还是 `ebase`。
- 壳层、左栏、后端配置边界是否判清。
- 参考页是否真的同构。
- 字段、状态、权限、跳转参数来源和待确认项是否写清。
- Mock 与真实接口边界是否隔离。
- 是否检查 XSS、越权、URL 拼接、敏感信息、上传下载、重复提交。
- 本次改动是否影响公共样式、公共左栏或同类页面。
