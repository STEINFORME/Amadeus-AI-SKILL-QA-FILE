# 热修和 deploy 流程摘要

## 适用范围

用于 LQTedu 前端项目分支走热修时的 Git、deploy 目录和发布提醒流程。

## 核心结论

- 热修分支必须从最新 `master` 建，不要从项目分支建。
- 项目分支推送或确认已和远端一致后，必须重建部署包目录。
- `deploy/` 只放本次热修要交付的前端文件，默认只收 `web/**`。
- 热修完成后的提醒对象是项目分支：提醒发布人将远端项目分支合并进 `master` 后删除项目分支。

## 标准流程

1. 确认当前项目分支已经推送到远端。
2. 重建 `deploy/` 部署包目录。
3. 从最新 `master` 建热修分支，例如 `hotfix/260508`。
4. 推送热修分支到远端。
5. 将远端项目分支合并到热修分支。
6. 推送合并后的热修分支。
7. 提醒发布人后续处理项目分支。

## deploy 重建规则

- 清空仓库根目录 `deploy/` 属于删除操作，必须先确认目标路径。
- 根据 `origin/master..origin/<projectBranch>` 的前端变更清单，把文件从项目分支按原路径导出到 `deploy/`。
- 示例：源文件 `web/oa/data/dataManager.jsp` 放到 `deploy/web/oa/data/dataManager.jsp`。
- 后端、配置、删除类变更不要伪装成前端部署文件，需要单独说明。
- 如果 diff 看起来异常大，应先计算 merge-base，确认比较基准。

## 硬性暂停

以下操作前必须确认：

- 清空 `deploy/`。
- `git push`。
- 删除远端分支。
- `git reset --hard`。
- 强制覆盖。
- 任何可能导致数据丢失的命令。

## 验证清单

- `master` 和 `origin/master` 指向一致。
- 项目分支和远端项目分支指向一致。
- `deploy/` 已清空并重新生成。
- `deploy/` 只包含本次热修前端文件且路径保持 `deploy/web/...`。
- 热修分支从 `master` 创建，不是从项目分支创建。
- 合并后本地和远端热修分支都包含项目分支修复提交。
- `git status --short --branch` 没有出现本次热修不该携带的已暂存或已修改文件。
- 最终提醒写清楚：远端项目分支合并进 `master` 后删除远端项目分支。

## 示例提醒

正确：

```text
origin/912 已经合并进 origin/hotfix/260508。
接下来请发布人在远端将 912 合并到 master，完成后删除远端 912 分支。
```

不要默认写：

```text
发布后删除 hotfix/260508。
```
